# ADR 0012: Prisma ORM for Writes; SQL Views for Analytics

- **Date:** 2026-03-24
- **Author:** Joe Peters
- **Status:** Accepted

## Context

Trapeze has two distinct database access patterns:

1. **Writes (ingest, sync, infer):** Upsert individual records with well-defined schemas. Type safety and schema evolution tooling are valuable. Examples: `prisma.testCase.upsert(...)`, `prisma.ciRun.create(...)`.

2. **Reads (Metabase dashboards, flake analysis):** Complex aggregations across multiple tables — coverage percentages, test trend windows, link confidence rollups, flake scores. These benefit from optimized, handwritten SQL and Postgres-specific features (window functions, `DISTINCT ON`, `jsonb_agg`).

Options considered:

1. **Raw SQL everywhere** — maximum control; no type safety; boilerplate for simple inserts
2. **ORM everywhere** — type-safe; ORM-generated SQL sometimes suboptimal for complex analytics
3. **ORM for writes + SQL views for reads** — each tool applied where it is strongest
4. **GraphQL layer** — adds resolver complexity; unnecessary for a backend-only ETL system

## Decision

**Prisma ORM for all writes and simple reads.** Schema is the single source of truth; migrations are generated by `prisma migrate dev` and never hand-edited. `db:generate` must be run after any schema change to regenerate the TypeScript client.

**Handwritten SQL views for analytics,** applied via `scripts/apply-views.ts` (runs `CREATE OR REPLACE VIEW` statements from `prisma/views/`). Views expose natural "table-like" surfaces for Metabase without requiring custom SQL in every dashboard question.

The 20+ views include: coverage rollup by feature area, test trend windows (7-day rolling), link governance quality scores, flake detection results, and execution health summaries.

## Consequences

**Positive:**

- **Type safety on writes:** Prisma generates TypeScript types from the schema; schema mismatches are caught at compile time, not at runtime
- **Migration tooling:** `prisma migrate dev` generates SQL diffs automatically; no manual migration writing for schema changes
- **Metabase integration:** views appear as natural tables in Metabase's data browser; analysts build dashboards without writing SQL
- **Analytics optimization:** window functions, `DISTINCT ON`, `jsonb_agg`, and other Postgres-specific features are available in view SQL without ORM limitations
- **Developer experience:** `npm run etl:sync:jira` reads cleanly without raw SQL scattered through ingest scripts

**Negative:**

- **Dual maintenance:** adding a new metric requires both a Prisma schema migration (for the source data) and a view SQL update (for the aggregation)
- **View drift:** if a schema migration renames a column, views that reference the old column name will break until updated
- **No Prisma types for views:** views are not in `schema.prisma`; reading from a view requires raw SQL (`prisma.$queryRaw`) with manual typing

**Mitigations:**

- Schema migrations and view updates are applied together in `npm run db:migrate && npm run db:views` — a single command covers both
- The `.claude/agents/prisma-schema-reviewer.md` agent reviews schema changes for view impact before migration creation
- `apply-views.ts` uses `CREATE OR REPLACE VIEW` (idempotent); views can be safely re-applied after any schema change
